The Night I Realized Python Has Been Lying to Me

It's 12:30 AM. I've been coding in Python for two days. Lists work. Dictionaries work. Functions work. So why do I suddenly need something called classes?

That's when my brain hit the wall.

I wasn't thinking deeply enough about what Python actually is. I thought classes were an optional advanced feature. Turns out, classes are the foundation of everything in Python. I'd just been using pre-built ones without realizing it.

By 3 AM, I understood. This post documents that journey.

Part 1: The Shocking Truth — Everything Is an Object

Here's something that should have broken my brain on Day 1:

Every single value in Python is an object.


x = 5          # x is an integer OBJECT
name = "Ali"   # name is a string OBJECT
data = [1,2,3] # data is a list OBJECT

Not variables. Objects.

What does that mean?

It means every value has:

  1. State — The data it contains
  2. Methods — Things it can do

# String object has methods
name = "Muhammad"
print(name.upper())        # String knows how to uppercase itself
print(name.lower())        # String knows how to lowercase itself
print(name.replace("m", "M"))  # String knows how to replace characters

The string "Muhammad" isn't just data sitting there. It's an intelligent object that knows how to manipulate itself.


# List object has methods
numbers = [3, 1, 4, 1, 5]
numbers.append(9)  # List knows how to add an item
numbers.sort()     # List knows how to sort itself
print(numbers.count(1))  # List knows how to count occurrences

The revelation: I've been working with objects since Day 1. I just didn't know what they were called.

Part 2: State vs Methods — What Makes an Object Intelligent

Let's break this down with a real example: a car.

State: What the Car Knows About Itself

Every car has properties that define its current condition:

  • Color (blue, black, white)
  • Current speed (0 mph, 50 mph, 100 mph)
  • Fuel level (empty, half-full, full)
  • Position (latitude, longitude)

This is the state of the object.

Methods: What the Car Can Do

A car isn't just data. It has capabilities:

  • accelerate() — increase speed
  • brake() — decrease speed
  • turn_left() — change direction
  • refuel() — add fuel

This is the behavior of the object.

Two cars, same class, different states:

Car A:
- Color: Blue
- Speed: 50 mph
- Fuel: 3/4 full

Car B:
- Color: Black
- Speed: 0 mph
- Fuel: 1/4 full

Both cars can accelerate. Both can brake. But their current states are different.

This is the essence of object-oriented programming: organizing data (state) with the functions that manipulate it (methods).

Part 3: Why You Can't Just Use Lists and Dictionaries

Python gives you lists and dictionaries out of the box. So why create your own classes?

Let me show you the problem.

The wrong way (using plain data):


# Representing a point without a class
point_1_x = 5
point_1_y = 10

point_2_x = 3
point_2_y = 7

# Now imagine you have 1000 points
# You'd have 2000 variables floating around
# Pure chaos.

The slightly better way (using dictionaries):


point_1 = {"x": 5, "y": 10}
point_2 = {"x": 3, "y": 7}

print(point_1["x"])  # 5

This works, but it's fragile:

  • No protection against typos ({"X": 5} is different from {"x": 5})
  • No built-in methods (you'd have to write distance calculation every time)
  • Mixing data with code creates confusion

The right way (using a class):


class Point:
    def __init__(self):
        self.x = 0
        self.y = 0

p = Point()
q = Point()

print(p.x)  # 0
print(q.x)  # 0

Now:

  • Each Point is organized and consistent
  • You can add methods to calculate distance, find midpoint, etc.
  • Code is clean and reusable

Part 4: Building Your First Class — The Point Class

Let's break down the simplest possible class:

First Class Example 

class Point:
    ''' Point class for representing x,y coordinates '''
    
    def __init__(self):
        ''' Constructor: runs when a new Point is created '''
        self.x = 0
        self.y = 0

p = Point()
q = Point()
print("p.x =", p.x, "q.x =", q.x)

What's Happening Here?

Line 1: class Point:

  • You're defining a new type of object called Point
  • From now on, Python knows what a "Point" is

Line 4: def __init__(self):

  • This is the constructor (initializer)
  • It runs automatically whenever you create a new Point
  • self means "the object being created"

Lines 6-7: Setting attributes


self.x = 0
self.y = 0
  • You're giving each Point object two attributes: x and y
  • Both start at 0 (the origin)

Creating Instances (Objects from the Class)


p = Point()  # Create first Point
q = Point()  # Create second Point

print(p.x)   # 0
print(q.x)   # 0

# Are they the same object?
print(p is q)  # False

The key insight: p and q are two different objects created from the same class blueprint.

They both have x and y attributes set to 0, but they are separate entities in memory.

If you change one, the other doesn't change:


p.x = 5
print(p.x)  # 5
print(q.x)  # 0 (q is unchanged)

Part 5: Understanding __init__ and Instantiation

The __init__ method is special. It's the constructor.

What is Instantiation?

Instantiation is the process of creating a new object from a class. It has two steps:

  1. Create a blank object — Python creates an empty instance
  2. Run __init__ — Python calls the constructor to set up the object

Behind the scenes:


# Step 1: Create blank instance
p = <blank Point object>

# Step 2: Run __init__ on that object
p.__init__()  # which sets p.x = 0 and p.y = 0

The result: p is now a fully initialized Point object with x=0 and y=0.

The Factory Metaphor

Think of a class as a factory:

  • The class is the blueprint/machinery
  • Each object is a product rolling off the production line
  • The __init__ method is quality control that sets up each product

Part 6: State, Behavior, and Why This Matters

Let's build on the Point class to show state and behavior together:

Methods Example 

class Point:
    def __init__(self):
        self.x = 0
        self.y = 0
    
    def move_right(self, distance):
        '''Move point right by changing x'''
        self.x = self.x + distance
    
    def move_up(self, distance):
        '''Move point up by changing y'''
        self.y = self.y + distance
    
    def get_position(self):
        '''Return current position'''
        return (self.x, self.y)

p = Point()
print(p.get_position())  # (0, 0)

p.move_right(5)
p.move_up(3)
print(p.get_position())  # (5, 3)

Now our Point has:

State: x and y coordinates

Methods (Behavior):

  • move_right() — changes state
  • move_up() — changes state
  • get_position() — reads state

The magic: The data (x, y) and the functions that manipulate it (move_right, move_up) are bundled together in one object. This is why classes exist: organization and safety.

Part 7: The Real-World Engineering Connection

In electrical engineering, you design circuits as templates.

A power supply circuit template has:

  • State: Voltage (5V, 12V, etc.), Current (in amps)
  • Behavior: Step up voltage, step down voltage, convert AC to DC

You design it once. You manufacture it 10,000 times. Each one is an instance of the same template.


class PowerSupply:
    def __init__(self):
        self.voltage = 0
        self.current = 0
    
    def set_voltage(self, v):
        self.voltage = v
    
    def set_current(self, i):
        self.current = i
    
    def get_power(self):
        return self.voltage * self.current

# Manufacture instances
ps1 = PowerSupply()  # Power supply 1
ps2 = PowerSupply()  # Power supply 2

ps1.set_voltage(5)
ps1.set_current(2)

ps2.set_voltage(12)
ps2.set_current(1)

print(ps1.get_power())  # 10 watts
print(ps2.get_power())  # 12 watts

Same design. Different configurations. Different real-world use cases.

Part 8: Common Mistakes (And How I Made Them)

Mistake 1: Forgetting What self Means


# WRONG - treating self like a regular variable
class Point:
    def __init__(self):
        x = 0  # This is a LOCAL variable, not an attribute!
        y = 0

p = Point()
print(p.x)  # ERROR! p doesn't have attribute x

RIGHT:


class Point:
    def __init__(self):
        self.x = 0  # This creates an ATTRIBUTE on the object
        self.y = 0

p = Point()
print(p.x)  # 0 - works!

The lesson: self.x creates an attribute. x creates a local variable. Don't confuse them.

Mistake 2: Assuming Different Objects Are the Same


p = Point()
q = Point()

print(p is q)  # False - they're different objects!
print(p == q)  # Also False - different objects have different identities

Mistake 3: Modifying One Object Thinking It Won't Affect Others


p = Point()
q = Point()

p.x = 5

print(p.x)  # 5
print(q.x)  # 0 - q is unaffected!

This isn't a mistake, but it's important to understand: changing one object's state doesn't affect others.

Mistake 4: Creating Classes When You Don't Need Them


# OVERKILL
class SingleNumber:
    def __init__(self, value):
        self.value = value

x = SingleNumber(5)
print(x.value)  # 5

# BETTER
x = 5  # Just use a number

Part 9: Why Every Value Is Really an Object

This is the key insight that clicked for me.

Python is built on classes. Everything you use is a class:


# int is a class
x = 5
print(type(x))  # <class 'int'>

# str is a class
name = "Ali"
print(type(name))  # <class 'str'>

# list is a class
numbers = [1, 2, 3]
print(type(numbers))  # <class 'list'>

When you call name.upper(), you're calling a method that exists on the string class.

When you call numbers.append(4), you're calling a method that exists on the list class.

Python comes with hundreds of pre-built classes. You can use them. Or you can create your own.

Common Questions

Q: Do I need to create a class for everything?
A: No. Use built-in types (int, str, list) when appropriate. Create a class when you need to organize related data and behavior.

Q: Why is it called __init__ and not init?
A: The double underscores tell Python "this is a special method." Other special methods include __str__, __len__, etc.

Q: Can I create multiple instances of the same class?
A: Yes. Each instance is independent with its own state.
